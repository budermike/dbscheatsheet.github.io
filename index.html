Unäre Operatoren
Projektion — πSpalten wählen
Wählt Teilmenge von Attributen aus. Entfernt Duplikate (Mengensemantik). Äquivalent zu SELECT in SQL.
π_name(student) π_name, date_of_birth(student)
Selektion — σZeilen filtern
Wählt Tupel die Prädikat p erfüllen. Vergleiche: =, <, ≤, >, ≥. Kombinieren mit ∧, ∨, ¬.
σ_{stud_no ≤ 3}(student) σ_{stud_no ≤ 3 ∧ date_of_birth > 1991-01-01}(student)
Umbenennung — ρNamenskonflikte lösen
Gibt Attributen neue Namen. Nötig wenn zwei Relationen Attribute mit gleichem Namen haben.
ρ_{a_no←pers_no, a_name←name}(assistant)
Joins
Kartesisches Produkt — ×
Kombiniert jedes Tupel von R mit jedem von S. Ergebnis: |R| × |S| Tupel. Meist sinnlos ohne Selektion.
lecture × professor
Theta-Join — ⋈_pShortcut für σ_p(R×S)
Allgemeinster Join. Prädikat p kann beliebige Vergleichsoperatoren enthalten.
lecture ⋈_{prof_pers_no = pers_no} professor
Natural Join — ⋈
Equi-Join auf gleich benannten Attributen. Entfernt redundante Spalten. Umbenennung mit ρ nötig wenn Namen nicht passen.
ρ_{pers_no←prof_pers_no}(lecture) ⋈ professor
Outer Joins — ⟕ ⟖ ⟗
Behält Tupel ohne Join-Partner, füllt fehlende Werte mit NULL auf.
L ⟕ R -- Left: alle Tupel von L L ⟖ R -- Right: alle Tupel von R L ⟗ R -- Full: alle Tupel beider
Semi-Join — ⋉ ⋊
Prüft welche Tupel einen Join-Partner haben — gibt aber nur die ursprüngliche Seite zurück (nicht verbunden).
L ⋉ R -- Tupel von L die einen Partner in R haben L ⋊ R -- Tupel von R die einen Partner in L haben
Mengenoperationen & Division
Union / Schnitt / Differenz
Beide Relationen müssen dasselbe Schema haben (gleiche Attributnamen und -typen).
prof1 ∪ prof2 -- alle Tupel aus beiden (Duplikate entfernt) prof1 ∩ prof2 -- nur Tupel in beiden prof1 \ prof2 -- nur Tupel in prof1, nicht in prof2
Division — ÷∀-Quantifizierung
Findet alle Werte in T die mit ALLEN Tupeln von R kombiniert werden können. Typisch für "für alle"-Queries.
attends ÷ lect1 -- "Alle Studenten die ALLE Lectures in lect1 besuchen" -- Formale Definition: T ÷ R = π_{T\R}(T) \ π_{T\R}( (π_{T\R}(T) × R) \ T )
Komplexes Beispiel
-- "Namen aller Studenten die Lecture 5 besuchen" π_name( student ⋈_{stud_no=stud_no} (σ_{course_no=5}(attends)) ) -- Operatoren kombinieren: Ergebnis ist immer eine Relation -- → kann als Eingabe für den nächsten Operator dienen
Grundstruktur
DRC-AusdruckDeklarativ
Spezifiziert was man will, nicht wie es berechnet wird. Variablen stehen für Domänenwerte (Attributwerte), nicht Tupel. Gleiche Ausdrucksstärke wie RA.
{ <x₁, x₂, ..., xₙ> | P(x₁, x₂, ..., xₙ) } -- links von |: freie Variablen (erscheinen im Ergebnis) -- rechts von |: Prädikat P das erfüllt sein muss
Bausteine
Atome — drei Formen
<x₁, x₂, ..., xₙ> ∈ r -- Tupel in Relation r x θ y -- Vergleich zweier Variablen (θ: =,≠,<,≤,>,≥) x θ c -- Vergleich Variable mit Konstante c
Quantoren & logische Operatoren
∃xExistenzquantormind. ein Wert für x erfüllt P
∀xAllquantoralle Werte für x erfüllen P
∧KonjunktionP₁ und P₂ müssen beide wahr sein
∨DisjunktionP₁ oder P₂ (mind. eines)
¬Negationnicht P

_Anonyme VariableWert wird nicht verwendet
Freie Variablen: nicht durch ∃ oder ∀ gebunden — nur diese erscheinen links von |
Beispiele
Selektion + Projektion
∃ bindet Variablen die nicht im Ergebnis erscheinen sollen (z.B. stud_no s).
-- "Namen & Geburtsdaten aller Studenten nach 1991" { <n, d> | ∃s( <s, n, d> ∈ student ∧ d > 1991-01-01 ) }
Join
Joins entstehen implizit: dieselbe Variable in zwei Relationen erzwingt gleiche Werte (= Join-Bedingung).
-- "Lecture-Titel mit zugehörigem Professorennamen" { <t, n> | ∃c, e, p( <c, t, e, p> ∈ lecture ∧ ∃o( <p, n, o> ∈ professor ) )}
Variable p erscheint in beiden Relationen → impliziter Equi-Join auf pers_no
Mehrere verschachtelte Bedingungen
-- "stud_no & Name aller Studenten die mind. 1 Prüfung -- bei Professor Curie hatten" { <s, n> | ∃d( <s, n, d> ∈ student ∧ ∃c, p, g( <s, c, p, g> ∈ exam ∧ ∃a, o( <p, a, o> ∈ professor ∧ a = "Curie" ) ) )}
Universelle Quantifizierung — ∀
Für "für alle"-Queries. ∀ kann als ¬∃¬ umgeschrieben werden. Implikation ⇒ häufig mit ∀ kombiniert.
-- "User bei denen alle Freunde mind. 1 Ski-Session haben" { UName | ∃UID( u(UID, UName, _) ∧ ∀FID( f(UID, FID) ⇒ ∃SID( s(SID, FID, _, _, 'Ski') ) ) )}
Sicherheit von Ausdrücken
Sichere vs. unsichere Ausdrücke
Ein Ausdruck ist sicher wenn er kein unendliches Ergebnis produzieren kann. Freie Variablen müssen range-restricted sein.
-- UNSICHER: alle Tupel NICHT in student → unendlich viele { <s, n, d> | ¬( <s, n, d> ∈ student ) } -- SICHER: s ist durch ersten Teil auf student-Werte beschränkt { <s> | <s, n, d> ∈ student ∧ ¬∃<s, c> ∈ lecture }
Grundstruktur
SELECT … FROM … WHERESFW-Block
Grundbaustein jeder SQL-Query. SQL entfernt Duplikate nicht automatisch (Bag-Semantik).
SELECT stud_no, name FROM student WHERE stud_no < 3;
DISTINCT
SELECT DISTINCT date_of_birth FROM student;
Filterung & Prädikate
BETWEEN / LIKE / NULL
WHERE date_of_birth BETWEEN '1992-01-01' AND '1994-01-01' WHERE name LIKE 'Sm%' -- % = beliebig viele Zeichen WHERE name LIKE '_mith' -- _ = genau ein Zeichen WHERE date_of_birth IS NULL -- NULL-Prüfung mit IS, nicht =
Joins & Mengenoperationen
JOIN (implizit)
SELECT * FROM student S, attends A, lecture L WHERE S.stud_no = A.stud_no AND A.course_no = L.course_no;
JOIN (explizit)
SELECT * FROM R [NATURAL | LEFT OUTER | RIGHT OUTER | FULL OUTER] JOIN S [ON R.A = S.B];
UNION / INTERSECT / EXCEPT
SELECT * FROM prof1 UNION SELECT * FROM prof2; SELECT * FROM prof1 INTERSECT SELECT * FROM prof2; SELECT * FROM prof1 EXCEPT SELECT * FROM prof2;
UNION ALL behält Duplikate. UNION entfernt sie automatisch.
Aggregation & Gruppierung
Aggregatfunktionen
SELECT count(*), count(DISTINCT x), sum(x), avg(x), max(x), min(x)
GROUP BY + HAVING
Alle nicht-aggregierten Attribute im SELECT müssen im GROUP BY stehen. HAVING filtert nach der Aggregation.
SELECT prof_pers_no, count(*) AS no_of_lect FROM lecture GROUP BY prof_pers_no HAVING count(*) > 3;
Sortierung, Subqueries & Views
ORDER BY / IN / EXISTS
ORDER BY date_of_birth DESC, name ASC; WHERE stud_no IN (SELECT stud_no FROM attends WHERE course_no = 5); WHERE EXISTS (SELECT * FROM assistant B WHERE B.mgr = P.pers_no);
CREATE VIEW / WITH / WITH RECURSIVE
CREATE VIEW myView AS SELECT ... ; WITH tmp AS (SELECT ...) SELECT * FROM tmp; WITH RECURSIVE r(a) AS ( SELECT ... -- initial UNION SELECT ... -- rekursiv (läuft bis keine neuen Zeilen) ) SELECT * FROM r;
INSERT
INSERT INTO professor VALUES(123456, 'Kossmann', 012); INSERT INTO professor(pers_no, name) VALUES(123456, 'Kossmann'); -- nicht angegebene Attribute → NULL (Key muss immer angegeben!) INSERT INTO professor(pers_no, name) SELECT pers_no, name FROM assistant WHERE pers_no = 111111;
DELETE / UPDATE
DELETE FROM professor WHERE pers_no = 123456;
Ohne WHERE werden ALLE Tupel gelöscht!
UPDATE professor SET office_no = 121 WHERE pers_no = 123456;
CREATE TABLE + Constraints
CREATE TABLE professor ( pers_no integer NOT NULL PRIMARY KEY, name varchar(80) NOT NULL, office_no integer UNIQUE CHECK (office_no > 0), area varchar(200) DEFAULT 'computer science' );
